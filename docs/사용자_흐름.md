# 쉼표 정류장 — 사용자 흐름

> 파생 문서. 제품 결정은 `제품_기획.md`만 변경하며 이 문서는 화면·예외 흐름을 설명한다.

## 시민 기본 흐름

```text
공통 QR/웹앱
→ [위치 허용] 또는 [정류장 직접 찾기]
→ 주변 후보 중 정류장명·4자리 번호·방면 확인
→ 현재 도착 버스와 시설상태 확인
→ [이 정류장에서 불편한 점 말하기]
→ 음성 또는 큰 버튼 입력
→ AI 구조화 결과 확인·수정
→ 제출
→ 확인 절차 안내 + 필요 시 공식 민원 채널 이동
```

첫 화면에서 목적지, 로그인, 나이, 방면을 먼저 묻지 않는다. 정류장 후보가 하나라도 결과 화면에서 대상 정류장을 확인시킨다.

## 위치·정류장 식별 예외

| 상황 | 처리 |
|---|---|
| 위치 허용 | 반경 내 후보를 거리순으로 제시하되 자동 확정하지 않음 |
| 맞은편 후보가 겹침 | 노선·4자리 번호·방면·랜드마크를 함께 표시 |
| 실내·경로당에서 접속 | 현재 장소 또는 정류장명 직접 검색 |
| 위치 거부·GPS 실패 | 4자리 번호, 정류장명, 지도 선택 제공 |
| 버스번호만 입력 | 해당 노선의 주변 정류장 후보를 좁히되 단독 확정 금지 |

## 도착·시설 안내 상태

- 실시간 도착: 제공기관과 생성시각 표시
- 대체 정보: `배차정보 참고` 등 실제 성격을 표시하고 실시간이라 하지 않음
- API 실패: 도착정보를 숨기되 시설 안내와 제보는 계속 이용
- 시설: `있음/없음 확인/미확인/정보 충돌`과 출처·확인일 표시
- 맞은편 시설: 현재 승차 지점의 시설로 합치지 않음

## 불편 제보 흐름

```text
“버스 오는 화면이 며칠째 꺼져 있어요”
→ AI 초안: [버스정보안내기] [작동 불량] [며칠째]
→ 사용자: 대상 정류장과 내용 확인/수정
→ 서버: 원문 최소보관 + 구조화 결과 + 정류장 ID
→ AI/규칙: 공식 BIT 설치자료 대조 + 유사 제보 중복 묶기
→ 관리자: “설치됨·고장 가능성·현장확인 권고” 검토
```

제보 완료 문구는 `정류장 확인과 개선 검토 자료로 활용됩니다`로 한다. 공식 민원 접수와 답변이 필요하면 기존 춘천시 채널을 별도 안내한다.

## 행정 흐름

```text
신규 신호함
→ AI 구조화·중복 묶음
→ 공식자료 충돌/고장/시설 요구 분류
→ 근거 카드 검토
→ 승인·반려·현장조사 배정
→ 조사 결과 입력 및 사람 확정
→ 공공 데이터 갱신 후보 + 시민 화면 반영
```

AI 권고와 사람 확정은 화면 색상·라벨·감사로그로 명확히 분리한다.
