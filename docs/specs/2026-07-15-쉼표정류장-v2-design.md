# 쉼표 정류장 v2 — 확장 기능 설계 (실시간·시설필터·공유강화·목적지길찾기)

> 작성일 2026-07-15. v1(기본 앱) 위에 4개 기능을 얹는다. v1 설계: `docs/specs/2026-07-15-쉼표정류장-design.md`.
> 확정된 결정(사용자 승인): 시설표시=정류장 강조 / 도보=실제경로(폴백 직선) / 실시간=폴백먼저·키오면자동 / 기능4=풀버전(환승).

## 0. 핵심 접근성 원칙 (전 기능 공통 — 사용자 강조)

**"키보드 0"의 정확한 의미:** 키보드/검색 기능을 없애는 게 아니다. **키보드를 전혀 쓰지 않고도 모든 핵심 기능을 불편 없이 완결**할 수 있어야 한다. 타이핑은 "빠른 지름길(선택)"일 뿐, 필수 경로가 되면 안 된다.

- 공유 받기: QR 스캔 또는 링크 탭만으로 완결(주소 타이핑 불필요).
- 목적지 선택: 즐겨찾기 목록에서 **탭**으로 선택(검색 타이핑은 선택지로만 제공).
- 시설 선택: 큰 칩 **탭**.
- 모든 주요 액션 버튼 ≥48px, 폰트 ≥18px, 아이콘+한글 병기.

## 1. 범위 · 구조 (4 워크스트림 + 파이프라인)

```
파이프라인 추가:  노선정보 → routes.json (노선별 정류장 순서 그래프)
                 (+ TAGO 키 있으면) 정류장↔TAGO nodeid 매핑 → stops.json에 tagoNodeId 추가
A. 실시간 도착   arrivals.ts 확장 (TAGO 실시간 ↔ 배차 폴백)
B. 시설 필터·지도강조·도보경로   상단 필터칩 + 마커 강조 + OSRM 도보(폴백 직선)
C. 공유 강화     QR + Web Share API + 큰 버튼
D. 목적지 길찾기 즐겨찾기=목적지, 주변정류장+환승 경로탐색+도착정보
```

각 워크스트림은 독립 테스트 가능. 오프라인 하드 제약 유지(실시간·도보경로는 항상 폴백).

## 2. 데이터/파이프라인 변경

### 2.1 `app/public/data/routes.json` (신규)
노선정보(`노선번호, 노선(id), 정류장순서, 정류장(관리번호)`)에서 생성:
```ts
interface RoutesFile {
  generatedAt: string;
  routes: { routeId: string; routeNo: string; stops: string[]; }[]; // stops = 관리번호 순서대로
}
```
- 각 `노선(id)`를 한 방향으로 간주(정류장순서 오름차순). 이것이 기능 D의 도달성/방향 판정 그래프.

### 2.2 `stops.json`에 `tagoNodeId?` 추가 (TAGO 키 있을 때만, 빌드 단계)
- TAGO 버스정류소정보(도시코드 32010 춘천)로 전체 정류소(nodeid, gpslati/gpslong) 수집 → 우리 정류장 좌표와 **최근접(≤50m) 매핑**. 매칭 실패 정류장은 `tagoNodeId` 부재 → 런타임 폴백.
- 키 없으면 이 단계 skip(모두 폴백). **키는 `app/.env`의 `VITE_TAGO_KEY`. 빌드 스크립트가 없으면 그냥 건너뜀.**
- ⚠️ ID 체계 상이(승하차·그늘막과 동일한 위험) → 좌표 매핑의 오매칭 가능. 50m 초과·모호는 미매핑(폴백)으로 정직 처리.

## 3. A. 실시간 도착 (TAGO)

`app/src/lib/arrivals.ts` 확장:
- `getArrival(stop, routeNo?)`: `stop.tagoNodeId` && `VITE_TAGO_KEY` 있으면 TAGO 도착정보 API 호출(2.5s 타임아웃, AbortController). 응답 파싱 성공 시 `{text:"7분 후 (2번째 전)", live:true, byRoute:[...]}`. 실패/무키/미매핑 시 `{text:"배차간격 약 N분", live:false}`.
- 무한 스피너 금지. 로딩은 폴백 텍스트를 먼저 보여주고 실시간이 오면 교체.
- CSP `connect-src`에 `apis.data.go.kr` 유지(이미 허용됨).

## 4. B. 시설 필터 + 지도 강조 + 도보 경로

### 4.1 필터 (시민 홈 상단)
- 칩 3개: 그늘 / 의자 / 조명. 탭하면 토글. 선택된 시설이 **`status==="yes"`인 정류장만** 마커 강조(그 외는 흐리게/축소). 복수 선택 = AND.
- "미확인"은 강조 대상 아님(있음만). 정직성 유지.

### 4.2 도보 경로 (`app/src/lib/walking.ts`)
- `getWalkRoute(from, to): Promise<{minutes:number; polyline:[number,number][]; real:boolean}>`.
- OSRM foot: `https://router.project-osrm.org/route/v1/foot/{lng},{lat};{lng},{lat}?overview=full&geometries=geojson` (2.5s 타임아웃). 성공 시 실제 경로선+시간(`real:true`).
- 실패/오프라인 → 직선거리 haversine ÷ 80m/분, `real:false`, polyline=[from,to] 직선. 문구 "직선거리 약 N분".
- 마커/정류장 탭 시 지도에 경로선 그리고 카드에 "걸어서 약 N분" 표시.
- CSP `connect-src`에 `https://router.project-osrm.org` 추가(보안 에이전트).

## 5. C. 공유 강화 (키보드 없이)

`app/src/features/share/` 확장:
- **QR 코드**: 기존 `buildShareUrl(ids)` 결과를 QR로 렌더(경량 QR 라이브러리, 인라인/오프라인). 자녀가 "공유" → QR 표시 → 부모가 카메라로 스캔 → 링크 열림 → 즐겨찾기 자동 등록(기존 ImportOnLoad).
- **Web Share API**: `navigator.share` 지원 시 네이티브 공유시트(카톡 등)로 링크 전송. 미지원 시 QR + 링크 복사 버튼.
- 큰 "가족에게 공유" 버튼. 타이핑 0.

## 6. D. 목적지 길찾기 (풀, 환승 포함)

진입점: 홈의 큰 버튼 **"버스로 가기"**(잠정명, 도착안내기 시설과 구분). 탭 → 즐겨찾기(=목적지) 목록 → 목적지 탭.

### 6.1 엔진 `app/src/features/trip/planTrip.ts`
- 입력: 현위치, 목적지 stop, `routes.json`, `stops`.
- ① 현위치 주변 출발 후보 정류장 N개(도보 ≤ 사용자 반경, 도보시간 순).
- ② 도달성 판정: 각 출발 정류장에서 목적지 정류장으로 가는 경로.
  - **직행**: 같은 route에 출발<목적지 순서 → 직행.
  - **1환승**: 출발 route의 하위 정류장 중 하나가 목적지 route의 상위 정류장과 같은 정류장(또는 도보 인접) → 환승.
  - 탐색은 정류장-노선 그래프 BFS, **환승 ≤ 1~2회로 제한**(과탐색·복잡도 방지).
- 출력: `TripOption[]` = `{ boardStop, alightStop?, transferStop?, routeNos:[], walkMin, arrival(폴백/실시간) }`. 직행 우선 정렬.

### 6.2 화면 `app/src/features/trip/TripView.tsx`
- 목적지명 큰 제목 + 결과를 **큰 카드 리스트**: "○○ 정류장까지 걸어서 4분 → 7번 버스 (3분 후 도착) → 목적지". 한 화면 한 결정.
- 각 카드의 버스 도착은 A(실시간/폴백) 사용. 도보시간은 B 사용.
- 결과 없으면 "직접 가는 버스를 찾지 못했습니다"(정직). 

## 7. 정직성 · 제약 (v1 규칙 유지 + 추가)
- 실시간·도보경로·경로탐색은 네트워크/키 의존 → **항상 폴백**(배차간격/직선거리). 오프라인 데모 무중단.
- TAGO/OSRM 미가용은 화면에 "약", "직선거리 기준" 등으로 정직 표기. "실시간"이라고 거짓 표기 금지.
- 시설 강조는 "있음"만(미확인 제외). 합성 점수 없음(v1 규칙).
- 키보드 없이 모든 핵심 흐름 완결(0절).

## 8. 에이전트 (기존 7팀 재사용, 병렬)
- 데이터: routes.json + (키 있으면)TAGO 매핑 빌드
- 백엔드: arrivals(TAGO) + walking(OSRM) 연동·폴백
- 프론트: 필터칩·지도강조·도보경로선·QR/공유·목적지 길찾기 UI
- 디자인/접근성: 키보드리스 흐름·대형 카드·단순성 감수
- 보안: CSP에 OSRM 추가, TAGO 키 처리, QR 라이브러리 감사
- QA: 신규 흐름 4종 + 오프라인 폴백 검증

## 9. 테스트 전략
- 순수 로직 vitest: planTrip(직행/환승/미도달), getWalkRoute 폴백, arrivals 폴백, buildShareUrl/QR payload, 필터.
- 네트워크 의존(TAGO/OSRM 실호출)은 모킹 + 폴백 경로 검증. 실경로 육안은 QA 수동.
