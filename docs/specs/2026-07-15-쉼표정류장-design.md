# 쉼표 정류장 — 설계 문서 (본선 개발)

> 작성일 2026-07-15. 원본 계획: `fable/개발계획_Opus핸드오프.md`, 기획서: `fable/생활 접근성 개선 서비스 - 오댕국.pdf`.
> 이 문서는 핸드오프 계획을 실제 확보된 데이터에 맞춰 구현 가능한 설계로 확정한 것이다.

## 1. 목적 · 완료 기준

**"버스를 기다리는 자리가 살 만한지"를 알려주는 서비스.** 고령자 대상 시민앱(B2C)이 데모 주인공, 행정 대시보드(B2G)가 정책실효 담당.

**완료 = 아래 데모 시나리오가 네트워크 불안 상태에서도 통과** (핸드오프 8절):
1. 접속 → 3초 내 지도 + 가장 가까운 정류장 카드 (조작 0회)
2. 카드에서 그늘·의자·조명·안내기 **3상태** + 출처 배지 확인
3. 자녀 폰 → 공유 링크 → 부모 폰 즐겨찾기 자동 등록 시연
4. (P1) 배차 긴 노선에서 대안 정류장 제안 뜸 / 곧 도착이면 안 뜸
5. (P1) A4 안내문 인쇄 미리보기
6. 대시보드 여름 프리셋 → TOP 10 표 → CSV 다운로드

평가배점: 기술구현 40 / 정책실효 30 / 서비스성 20 / 데이터활용 10. **화면 완성도 최우선.**

## 2. 확보 데이터 · 조인 전략 (실측 검증 완료)

`data/` 폴더 실제 파일 기준. **두 개의 ID 세계가 존재**하며 직접 매칭이 불가능함을 실측으로 확인:

| 파일 | 키 | 좌표 | 용도 |
|---|---|---|---|
| 버스정류장 위치정보_20260326 | `관리번호`(250xxx), `정류장번호` | ✅ 위경도 | **마스터**(1890 정류장) |
| 버스정류장 노선정보_20260326 | `정류장`(=관리번호 250xxx) | ✅ | 경유노선 (관리번호 조인) |
| 버스노선별 시간대별 승하차_20251209 | `정류장아이디`(424xxxx) | ❌ | 시간대별 승차 수요 |
| 벤치 현황_20260601 | 없음(명칭+좌표) | ✅ | 의자 (공간매칭 30m) |
| 폭염대비접이식그늘막_20260610 | 없음(주소) | ❌ | 그늘 (지오코딩→공간매칭) |
| 가로등정보_20260610 | 없음(좌표) | ✅ | 야간조명 (반경 존재여부) |

**실측 결과:** `424xxxx ∩ 250xxx = 0`, `424xxxx ∩ 정류장번호 = 0` → ID 직접조인 불가. 이름 매칭은 95%(899중 859). 승하차는 이름 899개에 ID 1505개 → 양방향 정류장 다수.

**조인 전략:**
- **마스터**: 위치정보를 관리번호(250xxx) 기준 단일 진실. 이름 조인 안 함(핸드오프 C-1 준수).
- **노선**: 노선정보를 관리번호로 조인 → 각 정류장 경유노선 목록.
- **의자**: 벤치 좌표를 각 정류장 좌표에 최근접 매칭(≤30m) → "있음(벤치대장)". 매칭 없으면 "미확인"(없음 아님).
- **그늘**: 그늘막 주소를 빌드 시 1회 지오코딩(네트워크 필요, 결과는 캐시해 stops.json에 굽는다) → 최근접 정류장(≤30m) → "있음(그늘막대장)". 지오코딩 실패분은 미확인.
- **조명**: 정류장 반경(예 50m) 내 가로등 존재 → "있음(가로등대장)". 없으면 미확인(가로등 데이터가 전수라는 보장 없음 → 정직하게 미확인).
- **수요**: 승하차를 **정류장명(양방향 합산)** 으로 위치정보에 브리지. 같은 이름의 여러 마스터 정류장에 동일 수요를 부여하고, 화면·CSV에 "양방향 합산 기준" 명시. 이름 미매칭(5%)은 수요 "미확인".
- **로드뷰 조사(150곳)**: 팀이 채울 스프레드시트 양식을 파이프라인이 정의. 도착 시 정류장ID(관리번호) 기준으로 시설 3상태를 **덮어쓰기**(로드뷰가 대장보다 우선, 배지 "로드뷰 확인 (촬영 YYYY.MM)").

## 3. 아키텍처

```
[C] 파이프라인 (Python/pandas, 단일 재실행 스크립트)
        → public/data/stops.json  (앱·대시보드 공통 정적 입력, 단일 진실)
        → public/data/roadview_survey_template.csv (팀 조사 양식)
   ├→ [A] 시민앱  (React 라우트 /)
   └→ [B] 대시보드 (React 라우트 /admin)   ← 같은 앱, 서버 0개
```

**기술 스택 (핸드오프 제안에서 2가지 변경, 사용자 승인 완료):**
- 파이프라인: **Python + pandas**, 재실행 가능한 단일 스크립트. 출력은 `stops.json` 하나.
- 프론트: **Vite + React + TypeScript**. TS 채택 이유 = 3상태를 유니온 타입으로 강제해 "미확인을 없음으로 쓰는 코드 경로"를 컴파일 차단(핸드오프 A-2).
- 지도: **Leaflet + OSM**(키 불필요, 즉시 작동). *변경*: 핸드오프의 Kakao는 JS키 발급 필요 + 로드뷰는 앱 임베드가 아니라 팀 수동조사라 이점 없음. Kakao 키 확보 시 타일 레이어만 교체.
- 대시보드: **동일 React 앱의 `/admin` 라우트**. *변경*: 핸드오프의 Streamlit(Python 별도 서버) 대신 통합 → 서버 0개, 오프라인 작동, CSV는 브라우저 Blob, README 1개.
- 오프라인: PWA 서비스워커로 앱셸+stops.json+지도타일 캐시. "해커톤장 네트워크 없이 작동" 하드 제약 충족.

## 4. `stops.json` 스키마 (인터페이스 계약 — 세 에이전트 공통)

```ts
type Facility = "yes" | "no" | "unknown";        // "no"는 조사로 확인된 없음만. 기본은 "unknown".
type Source   = "roadview" | "bench_registry" | "shade_registry" | "light_registry" | "none";

interface FacilityInfo {
  status: Facility;
  source: Source;          // status가 yes/no일 때 근거. unknown이면 "none".
  capturedAt?: string;     // roadview일 때만 "YYYY.MM". 배지 문구에 사용.
}

interface Stop {
  id: string;              // 관리번호 250xxx (마스터 키)
  stopNo: string;          // 정류장번호(물리 표지판 번호)
  name: string;
  lat: number; lng: number;
  routes: string[];        // 경유 노선번호
  facilities: {
    shade: FacilityInfo;   // 그늘
    seat: FacilityInfo;    // 의자
    light: FacilityInfo;   // 야간조명
    sign: FacilityInfo;    // 도착안내기 (현재 소스 없음 → 전부 unknown, 로드뷰로만 채움)
  };
  demand?: {               // 승차 수요 (양방향 합산). 미매칭이면 필드 부재 = 미확인.
    byHour: number[];      // 24칸, 승차건수 합
    total: number;
    aggregatedBidirectional: boolean;  // 항상 true (표기용)
    matchedName: string;
  };
  headwayMin?: number;     // 배차간격(분) 캐시 — TAGO 실패 시 도착정보 폴백용
}
```

**불변식**: 조사/대장 근거 없이 `status:"no"`를 만드는 코드 경로 금지. 파이프라인·프론트 모두 기본값 `{status:"unknown", source:"none"}`.

## 5. 기능 범위

**P0 (필수)**
- A-1 첫 화면: 위치권한(거부 시 춘천 시청 기본좌표) → 지도+마커, 최근접 정류장 자동선택 → 하단 카드(정류장명 + 그늘/의자/조명/안내기 4아이콘 3상태).
- A-2 3상태 표기: 전 화면 공통. 출처 배지("로드뷰 확인(촬영 YYYY.MM)" / "○○대장 기준"). "현장 확인" 문구 금지.
- A-3 즐겨찾기 + 보호자 대리등록: localStorage 별표 저장. 공유 = 정류장ID를 쿼리스트링에 담은 URL 생성 → 부모 폰에서 열면 자동 즐겨찾기 + 대형 홈. 서버·계정 없음.
- B-1 대시보드: 조건 필터(예: 한낮 승차 상위 25% AND 그늘 미확인 AND 유개식 아님) → TOP N 표(정류장명·ID·승차량·시설·근거) → CSV 다운로드. **합성 점수 금지, 조건 필터만.** 계절 프리셋 4종(저장된 필터 조합).
- C-1 파이프라인: 위 조인 전략으로 stops.json 생성 + 로드뷰 양식 정의.

**P1 (데모 강화)**
- A-4 도착정보: TAGO 키 없음 → "배차간격 약 N분" 캐시 폴백으로 구현(스피너 없음). 키 확보 시 실시간 교체.
- A-5 대안 정류장 제안: 4조건 전부 만족 시만 노출(①같은 노선 ②300m 이내 ③확인된 시설끼리 우위 ④도보≤배차/2). 하나라도 미확인이면 숨김. 곧 버스 오면 안 뜸.
- A-6 A4 안내문 인쇄: 정류장 선택 → 큰 글씨 A4 인쇄 화면(브라우저 인쇄).
- B-2 근거 카드: TOP N 행 클릭 → 후보 사유 한 장(승차 순위·시설 미비·로드뷰 캡처 자리).

**P2 (시간 남으면, 착수 안 함)**: 다크/고대비, 글자크기 버튼, 대시보드 지도뷰, 풍향 결합.

## 6. 표현 규칙 (화면·코드 공통, 핸드오프 7절)
1. 시설은 3상태만. 미확인≠없음. 2. 배지 "로드뷰 확인(촬영시점)", "현장 확인" 금지. 3. 쉘터 수치는 "2024년 발표 기준 264곳 추가 계획". 4. 수요 순위엔 "양방향 합산 기준" 병기. 5. "춘천만의 희귀 데이터" 주장 금지. 6. 4일 표본은 "순위 상관 0.91~0.93" 근거로 방어. 7. 노선 단위 주장 금지(정류장 단위 수요만). 8. 합성 점수 화면표시 금지.

## 8. 리스크 · 폴백
- **엑셀 날짜 오염**: 승하차 노선번호 32개가 날짜로 변질("3-1"→"03월 01일") → 복원 매핑. 단, 조인은 정류장 기준이라 노선번호는 표시용.
- **지오코딩 네트워크**: 빌드 시 1회만, 결과를 stops.json에 구워 데모는 오프라인.
- **단계적 결합**: 좌표지도 → 노선 → 시설 → 로드뷰 순. 뒤 단계 실패해도 앞 단계로 데모 가능.
- **4일 표본**: 정류장 단위 수요로만 사용, 노선 주장 배제.
