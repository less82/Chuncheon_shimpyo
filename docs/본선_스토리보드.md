# 쉼표 정류장 — 본선 화면 구성·스토리보드

> 사전 준비 산출물. 최종 웹·앱과 핵심 분석 결과를 미리 완성하지 않는다.
> 제품 기준: `docs/제품_기획.md`.

## 1. 화면 구조

```text
정류장 QR
└─ 시민 모드
   ├─ C1 현재 정류장 확인
   ├─ C2 목적지 입력·확인
   ├─ C3 버스·대기환경 결과
   ├─ C4 정보 제보
   └─ C5 즐겨찾기

/admin
└─ 행정 모드
   ├─ A1 대기환경 현황
   ├─ A2 조사 검토 순서
   ├─ A3 정류장 근거 카드
   ├─ A4 조사 결과 반영
   └─ A5 시설별 개선 검토
```

## 2. 시민 화면

### C1. 현재 정류장 확인

목표: 맞은편과 혼동하지 않고 출발 정류장을 확정한다.

```text
┌──────────────────────────┐
│ 현재 버스를 탈 정류장       │
│ 석사극동아파트              │
│ 2122 · 강원대학교 방면       │
│                          │
│ [맞아요, 목적지 말하기]       │
│ [정류장이 달라요]            │
└──────────────────────────┘
```

필수 데이터: 관리번호, 4자리 번호, 현행명, 방면, 좌표, 운영상태.

### C2. 목적지 입력·확인

목표: 타이핑 없이 목적지를 입력하되 오인식을 사용자가 확정한다.

```text
┌──────────────────────────┐
│ 어디로 가세요?              │
│ [🎙 목적지 말하기]           │
│                          │
│ “강원대학교 병원”이 맞나요?   │
│ [맞아요] [다시 말하기]        │
│ [병원] [시장] [복지관]        │
└──────────────────────────┘
```

### C3. 버스·대기환경 결과 — 데모의 주인공

목표: `얼마나`와 `어떻게` 기다릴지를 한 화면에서 답한다.

```text
┌──────────────────────────┐
│ 강원대학교병원 가는 버스      │
│ 5번 · 약 17분 후              │
│ 실시간/TAGO · 14:21 생성      │
│                          │
│ 이 정류장의 대기환경          │
│ 🪑 의자       있음            │
│ ☂ 쉘터       없음 확인        │
│ 🌳 그늘       미확인           │
│ 💡 조명       있음            │
│ 🖥 BIT        없음            │
│ [근거·기준일 보기]             │
│                          │
│ 지금은 한낮이고 대기시간이     │
│ 깁니다. 그늘 상태는 미확인입니다.│
│                          │
│ [정보가 달라요] [즐겨찾기 ☆]   │
└──────────────────────────┘
```

금지: 맞은편 의자 이용 권고, 미확인을 없음으로 표시, 폴백을 실시간으로 표시.

### C4. 구조화 제보

```text
┌──────────────────────────┐
│ 어떤 정보가 다른가요?         │
│ [의자가 있어요]               │
│ [의자가 없어졌어요]            │
│ [쉘터가 있어요]               │
│ [BIT가 작동하지 않아요]        │
│                          │
│ 제보는 확인 후 반영됩니다.      │
└──────────────────────────┘
```

### C5. 선택적 즐겨찾기·데이터 동의

- 즐겨찾기: 기기 저장, 계정 없음.
- 선택 동의: `정류장·시간구간·연령대·목적지 범주를 시설 개선 통계로 제공`.
- 거부해도 안내 기능 동일.

## 3. 행정 화면

### A1. 대기환경 현황

```text
┌──────────────────────────────────────┐
│ 춘천 정류장 대기환경                  │
│ 운영 정류장 | 시설 미확인 | 시민 재확인 │
│                                      │
│ [여름 폭염] [겨울 한파] [야간] [BIT]    │
└──────────────────────────────────────┘
```

KPI는 데이터가 실제로 존재할 때만 표시한다. `시설 없음`과 `미확인`을 분리한다.

### A2. 조사 검토 순서

| 순서 | 방향별 정류장 | 공식 시간대 수요 | 시설 미확인 | QR 목적지 조회 | 제보 | 조사 사유 |
|---:|---|---:|---|---:|---:|---|
| 1 | 예시 정류장·병원 방면 | 공식 연결값/미확인 | 그늘·쉘터 | 집계값 | 집계값 | 한낮 수요와 정보결측 |

시민신호는 공식수요와 합산하지 않고 별도 열로 둔다.

### A3. 정류장 근거 카드

- 개별 관리번호·번호·방면
- 시간대별 공식 승차 원값과 표본기간
- 시설별 모든 관측이력·출처·날짜·충돌
- QR 이용·목적지 범주·선택 연령대 집계
- 관련 시민 제보
- `왜 조사 대상인가` 문장

### A4. 조사 결과 반영

```text
조사 전: 그늘 미확인
→ 사람이 근거를 보고 있음/없음/미확인 확정
→ 조사 후: 그늘 없음 확인 또는 있음
```

미검수 시민제보와 AI 초안은 확정 상태를 만들지 않는다.

### A5. 시설별 개선 검토

| 시설 | 진입 조건 | 화면 문구 |
|---|---|---|
| 그늘·쉘터 | 없음 확인 + 한낮 수요·대기시간 | 폭염 대응 개선 검토 |
| 의자 | 없음 확인 + 수요·대기시간 + 생활목적 참고 | 의자 설치 검토 |
| 조명 | 없음·고장 확인 + 야간 운행·수요 | 야간조명 개선 검토 |
| BIT·QR | BIT 없음·고장 + 긴 대기·정보조회 | 정보접근 보완 검토 |

모든 행에 `현장 공사 적합성·예산 미검토`를 표시한다.

## 4. 데이터 프레임워크

```text
공식 정류장·노선 ─┐
공식 시설대장 ────┼→ 방향별 정류장 마스터
현장·로드뷰 관측 ─┘          │
                              ├→ 시민 대기환경 카드
공식 승하차(확정 연결만) ─────┤
시민 익명 집계 ──────────────┤
                              └→ 조사/개선 검토 뷰
```

테이블 계약:

- `stops`: 방향별 정류장 마스터
- `facility_observations`: 시설 관측이력
- `official_demand`: 공식 연결 승차수요
- `citizen_event_aggregates`: 비식별 집계
- `survey_cases`: 조사 과업·검수 결과
- `improvement_candidates`: 파생된 시설별 개선 검토 뷰

## 5. 본선 현장 개발 프레임

### 사전 준비 완료 표시 가능

- 이 스토리보드와 와이어프레임
- 데이터 원본 목록·ID 오류 목록
- API·오픈소스 후보
- 스키마와 검증 규칙
- 발표자료 목차

### 현장에서 커밋으로 남길 것

1. 시민 C1~C3 최종 연결
2. C4 제보와 선택 동의 이벤트
3. A1~A3 조사 검토 화면
4. A4 조사 반영 전후 처리
5. A5 시설별 개선 검토
6. 실제 데이터 분석 결과·차트
7. 실기기·접근성·네트워크 검증

발표에는 `사전 준비`, `본선 신규 구현`, `기존 범용 코드 재사용`을 구분한 커밋·작업표를 한 장으로 공개한다.
