# 쉼표 정류장 — 제품 기획 기준서

> 이 문서는 제품 정의의 단일 기준(SSOT)이다. 전략·화면·데이터·테스트 문서가 충돌하면 이 문서를 우선한다.
> 한 변경은 `문제 → 대상 사용자 → 사용자 흐름 → 필요한 데이터 → 검증`을 함께 수정해야 완료된 것으로 본다.

## 1. 제품 한 문장

**BIT가 없거나 사용하기 어려운 춘천 버스정류장에서, 정류장 QR로 출발지를 자동 식별하고 고령 이용자가 목적지를 말하면 탈 수 있는 버스와 도착예정시간을 큰 글자로 보여주는 무설치 웹앱이다.**

QR은 음성정보를 제공하는 장치가 아니다. QR의 역할은 앱을 열고 `출발 정류장 ID`를 전달하는 것뿐이다. 음성은 앱의 목적지 입력 수단이며, 버스·도착정보는 앱 화면이 제공한다. 앱이 결과를 소리로 읽는 기능(TTS)은 현재 범위가 아니다.

## 2. 해결할 문제와 대상 사용자

### 핵심 문제

정류장에 도착한 이용자가 목적지는 알지만 다음 두 질문에 답하기 어렵다.

1. 이 정류장에서 목적지로 가려면 어떤 버스를 타야 하는가?
2. 그 버스는 언제 오는가?

BIT 미설치는 이 정보 접근 문제를 키우는 조건이지, QR 필요성 자체를 증명하는 인과 근거는 아니다. BIT가 있어도 작은 글자·복잡한 노선표·검색 타이핑 때문에 이용이 어려울 수 있다.

### 1차 대상

- 버스는 이용하지만 스마트폰 타이핑과 복잡한 지도 검색이 어려운 65세 이상 이용자
- 정류장 현장에서 목적지를 기준으로 즉시 탑승 판단을 해야 하는 사람
- 스마트폰 기본 카메라로 QR을 열고 마이크 권한을 허용할 수 있는 사람

보호자·요양보호사는 보조 사용자다. 춘천시는 QR 부착, 데이터 갱신, 미설치 BIT 정류장 보완을 담당하는 운영 사용자다.

## 3. 핵심 사용자 흐름

```text
정류장 QR 촬영
  → 웹앱 실행
  → QR의 관리번호로 현재 출발 정류장 확정
  → 정류장명·방면을 큰 글자로 확인
  → “목적지 말하기”
  → 목적지 후보 확인
  → 이 정류장에서 목적지까지 가는 버스만 표시
  → 노선별 도착예정시간·직행/환승·방면 표시
```

실패 상태도 제품 흐름이다.

- 음성 인식 불가: 큰 글자 직접 입력과 대표 목적지 버튼 제공
- QR의 정류장 ID가 폐기·불명: 가까운 현행 정류장으로 임의 대체하지 않고 오류 표시
- 실시간 도착정보 없음: `실시간` 표현을 쓰지 않고 시간표·배차간격 등 출처가 명확한 대체값만 표시
- 목적지까지 경로 없음: 다른 정류장·노선을 임의 생성하지 않고 찾지 못했다고 표시

## 4. 기능 우선순위

### P0 — 대상 시연의 본체

1. 정류장별 QR 딥링크와 출발 정류장 고정
2. 현재 정류장명·4자리 번호·방면 확인
3. 음성 목적지 입력과 인식 결과 확인
4. 해당 정류장에서 목적지로 가는 현행 노선 계산
5. 노선별 도착예정시간과 데이터 상태 표시
6. 고령자용 큰 글자·큰 버튼·한 화면 한 결정
7. 오류·권한 거부·네트워크 불안 폴백

### P1 — 행정 적용과 신뢰성

1. BIT 미설치·정보취약 정류장의 QR 배포 검토 목록
2. QR 부착·점검·교체 이력
3. 정류장·노선·도착 데이터의 기준일과 출처 관리
4. 고령 사용자 실기기 테스트 결과

### P2 — 보조 가치

- 의자·그늘·조명 등 대기환경 표시
- 즐겨찾기·보호자 공유
- 시설 조사와 설치 검토

P2는 P0 설명을 방해하면 발표와 첫 화면에서 제외한다.

| P2 | 시설 현황 | 관리번호, 시설별 3상태, 출처, 촬영·확인일 | 대기환경 보조정보 | 미확인 표시 |

### 사용하지 않는 근거

- 개별 관리번호와 공식 대응되지 않은 `424…` 승차 ID의 한낮 승차량
- 동명 정류장 양방향 합산값을 개별 정류장 우선순위로 사용한 값
- 행정동 고령인구를 해당 동의 모든 정류장 위험도로 대입한 값
- 카카오 지도에 보인다는 사실만으로 운영 중이라고 확정한 값

한낮 승차 원본은 도시 전체의 시간대 수요를 설명하는 참고자료로 보존하되, 공식 ID 대응표를 확보하기 전까지 개별 QR 배포 순위에는 사용하지 않는다.

## 6. 대상(大賞)을 위한 증거

기능 수보다 다음 네 증거가 중요하다.

1. **현장 문제 증거:** BIT 유무와 무관하게 기존 방식으로 목적지 버스·도착시간을 찾는 데 어려움이 있었는지 인터뷰·관찰
2. **데이터 정확성:** QR의 정류장, 노선, 방면, 도착정보가 현행 공식 데이터와 일치하는지 표본 대조
3. **사용자 효과:** 65세 이상 사용자가 도움 없이 QR부터 버스 선택까지 완료하는지 전후 비교
4. **행정 실행성:** 정류장별 QR 생성·부착·점검 비용과 담당 업무가 현실적인지 춘천시 관점 검토

### 최소 성공지표

- QR 촬영 후 올바른 출발 정류장 표시: 표본 100%
- 안내한 노선이 실제 정차·진행방향과 일치: 표본 100%
- 도착정보의 출처·생성시각 표시: 100%
- 고령 사용자 핵심 태스크 완료율, 평균 소요시간, 도움 요청률, 오조작 수 공개
- 잘못된 데이터일 때 조용히 추정하지 않고 오류·미확인으로 중단: 100%

## 7. 문서 원자성 규칙

- 제품 정의는 이 문서에서만 바꾼다. 다른 문서는 정의를 복사하지 않고 이 문서를 참조한다.
- QR 종류는 `정류장 진입 QR`과 `가족 공유 QR`을 이름·URL·화면에서 분리한다.
- 기능을 추가하거나 목적을 바꿀 때는 같은 커밋에서 이 문서, 사용자 흐름, 테스트, 관련 화면을 함께 갱신한다.
- 검증 전 기능은 `구현됨`이 아니라 `가설` 또는 `검증 필요`로 표시한다.
- 과거 계획과 반박 문서는 기록용이며 현재 요구사항의 근거로 사용하지 않는다.

## 8. 현재 구현과의 차이

| 항목 | 현재 상태 | 다음 작업 |
|---|---|---|
| QR로 출발 관리번호 전달 | 구현됨 | 현장 QR 표본 검수 |
| 음성 목적지 입력 | 구현됨 | 인식 결과 확인 단계를 분리 |
| 출발 정류장 표시 | 정류장명 중심 | 4자리 번호와 방면 추가 |
| 목적지 후보 확인 | 미흡 | 동명·유사명 후보 선택 화면 추가 |
| 노선 경로 계산 | 구현됨 | 현행 정차순서·방면 표본 검산 |
| 노선별 도착정보 | 구현돼 있으나 폴백 포함 | 실시간 출처·생성시각·폴백 종류 표시 |
| 대표 목적지 버튼 | 미구현 | 음성 미지원·권한 거부 대안으로 추가 |
| BIT 기반 QR 배포 목록 | 미구현 | BIT 파일과 현행 정류장 마스터의 ID 연결부터 검증 |
| 고령 사용자 효과 | 검증 전 | `docs/사용자_테스트.md` 수행 |

따라서 현재 앱은 핵심 흐름의 프로토타입이며, 데이터 정확성과 실제 사용자 효과까지 증명된 완성 서비스로 표현하지 않는다.
