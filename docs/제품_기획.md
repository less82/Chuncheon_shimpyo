# 쉼표 정류장 — 제품 기획 기준서

> 제품 정의의 단일 기준(SSOT). 전략·화면·데이터·테스트가 충돌하면 이 문서를 우선한다.
> 1차 개발기획서의 `사계절 대기환경 안내 + 시민 안내 + 행정 개선` 골조를 변경하지 않는다.

## 1. 제품 한 문장

**쉼표 정류장은 고령 이용자가 버스를 기다리는 시간과 현재 승차 정류장의 의자·그늘·쉘터·조명·BIT 상태를 쉽게 확인하고, 방향별 QR 이용과 시설 제보를 행정의 조사·개선 검토로 연결하는 사계절 버스정류장 대기환경 서비스다.**

기존 버스 앱이 `언제 오는가`를 알려준다면 쉼표 정류장은 `그동안 어떤 환경에서 기다리는가`까지 답한다.

## 2. 제품의 중심과 보조 기능

| 구분 | 역할 |
|---|---|
| 제품 중심 | 개별 방향 정류장의 대기시간·시설상태 안내 |
| 시민 접점 | 정류장 QR, 큰 글자, 음성 목적지 입력, 선택적 즐겨찾기 |
| 시민→행정 | 동의한 QR 이용·목적지 조회·시설 제보의 익명 집계 |
| 행정 중심 | 시설 미확인 정류장 조사 → 부재 확인 후 개선 검토 |
| 보조 | 노선·도착정보, 가족 공유, A4 안내문 |

QR은 음성 안내기가 아니다. QR은 앱을 열고 방향별 출발 정류장을 식별한다. 음성은 목적지 입력 방식이고, 도착정보는 예상 대기시간을 계산하는 보조 데이터다.

## 3. 대상 사용자

- 1차: 폭염·한파와 긴 대기에 취약한 65세 이상 버스 이용자
- 2차: 가족·요양보호사·복지기관 종사자
- 운영: 춘천시 교통·안전·복지·시설 담당자

가족의 대리등록을 기본 전제로 삼지 않는다. 고령 이용자가 집 근처 정류장 QR로 직접 시작하고, 즐겨찾기는 선택적으로 쉽게 설정한다.

## 4. 정책 업무 흐름

```text
[공식 정류장·노선·시설·수요]
            +
[동의한 시민 QR 이용·시설 제보]
            ↓
1단계 조사 검토 순서
  시설 미확인 + 방향별 수요/이용신호 + 계절·시간대
            ↓ 사람 확인
2단계 시설별 개선 검토 순서
  시설 없음 확인 + 수요 + 대기시간 + 정책 조건
            ↓ 별도 현장·법적·예산 검토
3단계 공사 여부 결정(서비스가 자동 결정하지 않음)
            ↓
시설 갱신 → 시민 화면 반영
```

## 5. 시설 안내 원칙

- 분석 단위는 맞은편과 합치지 않은 `방향별 개별 정류장`이다.
- 정류장 내부 시설, 인근 시설, 맞은편 시설을 분리한다.
- 맞은편 시설로 건너가 기다리라고 권하지 않는다.
- 상태는 `있음/없음 확인/미확인/정보 충돌`로 관리한다.
- 공식대장이나 반경 검색에 없다는 이유만으로 `없음`을 만들지 않는다.
- 시민 제보는 즉시 현재 상태를 바꾸지 않고 검수 대기 관측으로 저장한다.

## 6. 데이터 계층

### 공식·관측 데이터

| 계층 | 핵심 필드 | 용도 |
|---|---|---|
| 정류장 마스터 | 관리번호, 4자리 번호, 명칭, 좌표, 방면, 운영상태, 적용기간 | 방향별 분석 기준 |
| 노선·도착 | 노선, 정차순서, 방면, TAGO ID, 도착예정, 생성시각 | 목적지 경로·대기시간 |
| 시설 관측이력 | 관리번호, 시설종류, 상태, 위치관계, 출처, 확인일, 검수상태 | 시설 근거 보존 |
| 공식 수요 | 개별 승차 ID, 관리번호 대응, 시간대, 승차인원, 표본기간 | 공식 연결 행만 정류장 수요로 사용 |
| 시민 집계 | 관리번호, 시간구간, 선택 연령대, 목적지 범주, 행동, 집계건수 | 공공데이터 보조 신호 |
| 정책 산출물 | 조사사유, 시설별 개선사유, 데이터 결측, 공사검토 필요 | 행정 업무 전달 |

### 사용하지 않는 근거

- 동명 정류장 양방향 합산값을 개별 정류장 순위에 사용
- 맞은편 정류장의 수요를 반대 방향에 이전
- 행정동 고령인구를 모든 정류장의 실제 고령 승차량으로 대입
- QR 조회를 실제 승차·하차로 표현
- 미확인을 시설 없음이나 공사 필요로 해석

## 7. 시민 신호와 개인정보

즐겨찾기는 기기 안에 저장한다. 서버 집계는 별도 동의를 받은 최소정보만 사용한다.

- 수집 가능: 방향별 관리번호, 넓은 시간구간, 선택 연령대, 목적지 범주, 시설 조회·제보
- 수집하지 않음: 이름, 전화번호, 생년월일, 집 주소, 지속 GPS, 장기 사용자 ID
- 표현: `서비스 이용·목적지 조회 신호`; `실제 승하차`라고 부르지 않음
- 공개: 주·월 단위 집계, 소수 조합 비공개, 원시 이벤트 단기 보관

## 8. 본선 핵심 범위

본선 시점에 맞춰 여름 폭염 사례 하나를 끝까지 증명한다.

1. QR로 방향별 정류장 확인
2. 목적지 입력 후 버스·예상 대기시간 확인
3. 현재 정류장의 그늘·쉘터·의자·BIT 상태 확인
4. 정보 오류 제보 또는 선택적 즐겨찾기
5. 행정 화면에서 시설 미확인 조사대상 확인
6. 검수 결과를 반영하면 시설별 개선 검토 목록과 시민 화면이 함께 갱신

겨울·야간은 같은 데이터 구조의 확장 프리셋으로만 보여준다.

## 9. 중단 조건

- 개별 정류장 ID·방면을 확정하지 못했는데 수요나 시설을 연결함
- 시민 조회를 승차인원으로 발표함
- 미확인을 시설 없음으로 처리함
- 보도폭·도로점용·전력·예산 없이 공사순위를 확정함
- 실제 사용자 검증 없이 고령자 편의를 확정적으로 주장함

## 10. 문서 원자성

- 제품 정의는 이 문서에서만 변경한다.
- 변경 시 사용자 흐름·테스트·스토리보드·관련 화면 계약을 같은 작업에서 갱신한다.
- 과거 계획 문서는 기록이며 신규 요구사항으로 사용하지 않는다.
- 구현 상태와 목표 상태를 구분한다.
