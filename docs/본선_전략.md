> 작성일 2026-07-16. 근거: 「2026년 춘천시 데이터 활용 해커톤 모집 공고」(춘천시 데이터활용.hwp) 본선 심사기준 + 현재 코드베이스/데이터 실측.
> 본선: **2026. 07. 31(금)~08. 01(토) 무박2일**, 춘천ICT벤처센터. 결과물 = **발표자료 + 웹/앱 플랫폼**, 발표 **팀당 14분**, 20팀 중 대상 1팀(상금 300만 원).

| 심사항목 | 심사내용 | 배점 |
|---|---|---|
| **정책 실효성 (30점)** | 지역 현안 적합성, 실제 행정 활용 가능성 | 15 |
| | 시민 체감 효과, 확장 가능성 | 15 |
| **기술 구현력 (40점)** | 데이터 처리구조, 분석의 타당성 | 20 |
| | 기능 구현 수준, 시연 가능성 | 20 |
| **서비스성 (20점)** | 사용자 관점의 편의성, 전달력, 화면구성, 활용 시나리오의 명확성 | 20 |
| **데이터 활용성 (10점)** | 춘천 관련 공공데이터의 활용 적절성 | 5 |

※ 평가지표는 추후 변동될 수 있음(공고 명기). 지정과제 중 우리 과제 = **"생활 접근성 개선 서비스"**.

**배점이 말해주는 것:**
- 최대 항목은 **기술 구현력 40점**이고 그중 절반이 "시연 가능성". → **현장에서 끊기지 않고 도는 데모**가 가장 큰 단일 득점원. (우리는 이미 PWA 오프라인 폴백을 갖춤 — 행사장 와이파이가 죽어도 시연 가능. 이걸 발표에서 명시적으로 언급할 것.)
- 정책 실효성 30점은 **B2G 없이는 만점이 어려움**. "행정 적용 가능성 15점"은 공무원이 실제로 쓸 화면·산출물을 보여줘야 함.
- 데이터 활용성은 10점뿐. 데이터 종류를 늘리는 것보다 **결합 구조의 타당성**(5점)을 설명하는 쪽이 효율적.

## 1. 방향성 평가: B2C + B2G 투트랙이 맞는가?

**결론: 방향은 맞다. 단, 두 가지를 수정해야 대상권이다.**

### 1-1. 투트랙이 맞는 이유 (배점 매핑)

| 트랙 | 직접 공략 배점 |
|---|---|
| B2C 시민앱 | 시민 체감 효과 15 + 서비스성 20 + 시연 가능성 일부 = **~40점대** |
| B2G 대시보드 | 행정 적용 가능성 15 + 분석 타당성 20 + 데이터 결합 구조 5 = **~40점대** |

한쪽만 하면 구조적으로 30점 안팎을 버리게 됨. 투트랙은 선택이 아니라 이 배점표의 요구사항.

### 1-2. 수정 ① — B2G의 "실시간"은 버려라 (정직성 + 감점 리스크)

우리 승하차 데이터는 **2025.6.25~28 나흘치 정적 데이터**(엑셀 65535행 절단, 농촌 223곳 수요 미확인)다. 이걸로 "실시간 수요 분석"을 주장하면:
- 심사위원(데이터 전문가 포함 가능)이 "데이터 출처가 뭐냐"고 묻는 순간 무너짐 — **분석 타당성 20점 직격**.
- 우리 팀 정체성인 정직성 규칙(근거 없는 주장 금지)과도 충돌.

**재프레임:** "실시간 예산 분석" → **"시간대별 수요 패턴 기반 투자 우선순위"**
- 시간대별(byHour 24칸) 수요 × 시설 결핍(3상태) × 노선 수를 결합한 **정적이지만 방어 가능한 우선순위 모델**.
- "실시간"이 필요하면 유일하게 진짜 실시간인 **TAGO 도착정보**만 그 단어를 쓰고, 나머지는 "최신 집계 기준 YYYY.MM" 배지로 표기.
- 오히려 이 정직함 자체를 발표 무기로: *"저희는 없는 데이터를 실시간이라고 부르지 않습니다. 미확인은 미확인으로 표시합니다"* — 행정 적용 가능성 심사에서 신뢰로 환산됨.

### 1-3. 수정 ② — 무박2일 범위 통제

본선은 사실상 **개발 가용 시간 ~15시간**(1일차 14:30~24:00 + 2일차 00:00~07:30). 신기능은 아래 §5의 우선순위 P0~P1까지만. 새 데이터 파이프라인 재설계 금지 — stops.json 계약은 그대로 두고 위에 얹는다.

## 2. B2C 앱 — "집에서 목적지까지, 앉아서 기다리는 길"

### 2-1. 타깃 세대: 고령자(65+) 단일 타깃 유지

- 근거 제시용 스토리: 춘천은 고령화 진행 지역, 폭염 속 그늘·의자 없는 정류장 대기가 실제 건강 리스크. "시민 체감 효과 15점"은 **누가, 언제, 얼마나 체감하는지**를 구체적 인물(페르소나)로 답해야 함.
- 타깃을 넓히면(전 세대) 화면이 복잡해지고 서비스성 20점("특정 사용자 관점의 편의성")이 오히려 깎임. **한 세대에 깊게**가 정답.
- 페르소나 예: *"후평동 78세 김OO 님, 병원 가는 길. 스마트폰은 카카오톡만 쓰심."*

### 2-2. 핵심 가치 제안 (사용자 질문 그대로)

> "주변 쉼터·정류장 → 도착지까지의 연결성 → **가장 편하고 안락하게** 가는 방법"

이를 한 문장 기능으로: **"편안 경로(쉼표 경로)" — 가장 빠른 길이 아니라, 그늘·의자가 있는 정류장을 경유해 서서 기다리는 시간을 최소화하는 길.**

### 2-3. 기능 구성 (기존 자산 → 본선 추가)

**이미 있음 (재사용, 발표 때 "구현 완료"로 시연):**
- 지도 + 시설 필터(그늘/의자/조명/도착안내기 3상태), 시설 근거 배지("벤치대장 기준", "로드뷰 확인 YYYY.MM")
- 정류장 카드(경유 노선, 시간대 수요), 즐겨찾기, 대체 정류장 힌트(AltStopHint)
- 도보 경로(OSRM WalkLayer) + 여정 계획(planTrip)
- TAGO 실시간 도착 + 배차간격 폴백, PWA 오프라인
- URL 공유 대리등록(자녀가 부모 폰에 즐겨찾기 심어주기), A4 인쇄 포스터

**본선에서 추가 (P0 = 필수):**
1. **편안 경로 추천** — 출발지→목적지 입력 시 후보 정류장들을 "편안 점수"로 정렬해 제시. 점수 = 의자(확인) + 그늘(확인) + 야간이면 조명 + 도보거리 패널티. 기존 planTrip/altStop 로직 확장이라 15시간 내 가능.
2. **"지금 기다리기 좋은 곳"** — 현재 시각 + 폭염 시간대(예: 13~16시)면 그늘 있는 정류장을 우선 표시. 시간대 수요(byHour)와 결합해 "지금 붐비는 정류장 → 옆 정류장 제안".
3. **음성/초대형 모드 마감** — 이미 고령자 UI 기반이 있으니, 데모용으로 글자 크기 토글·아이콘+한글 병기 일관성 점검(디자인-a11y 에이전트 활용).

**P1 (시간 남으면):**
- 도착 알림(진동/소리) — "버스 3분 전, 이제 일어나세요"
- 보호자 공유 강화: 여정을 QR로 자녀에게 전송

**만들지 말 것 (P-거절):** 회원가입/로그인, 챗봇, 전 세대용 모드 분기, 실내지도. 전부 15시간을 잡아먹고 배점에 안 붙음.

### 2-4. 화면 구성 (고령자 UX 원칙 포함)

```
[홈] ──── 큰 버튼 2개가 전부
 ├─ "🚌 버스 타러 가기"  → 가까운 정류장 1곳을 바로 크게 (선택지 최소화)
 └─ "⭐ 자주 가는 곳"     → 즐겨찾기 목록 (병원/시장/집)

[정류장 카드] ─ 한 화면 = 한 정보 원칙
 ├─ 도착정보: "16번 버스, 약 5분" (숫자 최대 크기, 실시간/배차폴백 구분 배지)
 ├─ 시설: 아이콘+한글 병기 "🪑 의자 있음 (벤치대장 기준)" / "❓ 그늘 미확인"
 └─ 버튼 1개: "여기로 가는 길 안내"

[편안 경로 (신규)] ─ 지도 최소화, 문장 중심
 ├─ "① 집에서 4분 걸어서 → ② ㅇㅇ정류장(의자·그늘 있음)에서 앉아서 기다리기
 │    → ③ 16번 버스 → ④ 병원 앞 하차"
 ├─ 비교 한 줄: "가장 가까운 정류장보다 1분 더 걷지만, 앉아서 기다립니다"
 └─ [크게 보기] [자녀에게 보내기(QR)]

[즐겨찾기] ─ 이름 크게, 사진/이모지, 탭 1번이면 경로 시작
```

**고령자 UX 체크리스트 (서비스성 20점 방어):**
- 글자 최소 20px/버튼 56px 이상, WCAG AA 대비, 터치 타깃 간격
- 화면당 의사결정 1개, 뒤로가기는 항상 같은 위치
- 전문용어 금지: "정류소 ID" ❌ → "정류장 번호판의 숫자" ⭕
- 모든 상태에 문장 폴백: 인터넷 끊김 → "지금은 저장된 정보로 보여드려요 (어제 기준)"

## 3. B2G 웹 — "예산을 어디에 먼저 쓸지, 근거와 함께"

### 3-1. 공무원(수요자)이 실제로 원하는 것

담당 부서를 상상하라: 교통행정과·도시재생과 주무관. 이들이 의회·상급자에게 보고할 때 필요한 것은 화려한 지도보다 **"방어 가능한 근거"**다.

| 공무원의 질문 | 우리가 주는 답 |
|---|---|
| "내년 벤치/그늘막 예산 N개, 어디부터?" | **투자 우선순위 랭킹** (수요×결핍 점수, 정렬된 목록) |
| "왜 거기가 1순위인가?" (의회 답변용) | **근거 카드** — 시간대 수요 그래프 + 시설 미비 현황 + 데이터 출처 명기 |
| "노인이 많이 쓰는 곳인가?" | 병원·복지관·시장 인접 정류장 가중치 (POI 결합) |
| "이거 보고서에 붙일 수 있나?" | **CSV 내보내기(이미 구현) + A4 인쇄(이미 구현)** — 한글(hwp) 문화권 행정엔 인쇄물이 곧 채택 |
| "데이터가 오래된 거 아닌가?" | 모든 지표에 기준일 배지 + "미확인 1,886곳 → 조사 필요 목록 자체가 산출물" |

### 3-2. 제공 데이터/지표 설계

**핵심 지표: 쉼표지수 (투자 우선순위 점수)** — 정류장별로:

```
쉼표지수 = 수요 가중치 (시간대별 승차, 특히 한낮 11~16시 폭염 시간대)
         × 시설 결핍도 (의자·그늘 '있음 확인' 없을수록 ↑)
         × 이용 취약 가중치 (병원/복지시설/시장 반경 300m 인접)
         × 노선 수 (환승·대기 발생량 프록시)
```

- 모든 항은 **이미 보유한 데이터**(stops.json + POI 지오코딩 소량 추가)로 계산 가능 → 무박2일 내 구현 현실적.
- "미확인" 시설은 결핍으로 단정하지 않고 **"조사 우선순위" 별도 트랙**으로 분리 — 정직성 유지가 곧 행정 신뢰성.

**대시보드 산출물 4종:**
1. **우선순위 지도** — 쉼표지수 히트맵/등급 마커, 시간대 슬라이더(아침/한낮/저녁 수요 전환)
2. **Top-N 리스트 + 근거 카드**(EvidenceCard 확장) — "1위 인성병원 앞: 한낮 승차 4위(양방향 합산), 의자 미확인, 노선 12개, 병원 도보 2분"
3. **예산 시뮬레이터(간단 버전)** — "벤치 1개 80만 원 × 예산 3,000만 원 → 37곳 커버, 추정 수혜 승차 N건/일" 슬라이더 1개면 충분. 심사위원 앞에서 움직이는 숫자는 강력함.
4. **조사 필요 목록** — 미확인 1,886곳 중 수요 상위 정류장부터 로드뷰/현장조사 우선순위 CSV. *"이 앱은 완성된 답이 아니라 행정이 계속 쓰는 도구"* 라는 확장성(15점) 서사.

### 3-3. B2G 화면 구성

```
[대시보드 홈]
 ├─ KPI 4개: 정류장 1,890 / 의자 확인 825 / 그늘 확인 4 / 미확인 조사대상 N
 ├─ 우선순위 지도 (시간대 슬라이더 + 시설 필터)
 └─ Top 20 우선순위 테이블 → 행 클릭 시 근거 카드

[근거 카드 (상세)]
 ├─ 시간대별 승차 그래프 (양방향 합산 기준 병기 — 정직성 규칙 §3)
 ├─ 시설 4종 3상태 + 출처 배지
 ├─ 인접 POI (병원·시장·복지관 거리)
 └─ [CSV 내보내기] [A4 보고서 인쇄]

[예산 시뮬레이터]
 └─ 예산 슬라이더 → 커버 정류장 수·수혜 승차 건수 실계산
```

### 3-4. 추가로 확보하면 좋은 공공데이터 (데이터 활용성 5+5점 보강)

| 데이터 | 용도 | 난이도 |
|---|---|---|
| 춘천시 행정동별 고령인구 통계 (KOSIS/공공데이터포털) | 취약 가중치 정밀화 | 낮음 — 정적 CSV 1개 |
| 병원·경로당·전통시장 POI (기보유 지오코딩 파이프라인 재사용) | 인접 가중치 | 낮음 |
| 주최측 요청 데이터 5건 (도착 시) | 보강 | 도착해야 판단 |

※ 새 데이터는 **결합 구조를 설명할 수 있는 것만** 추가. 개수 자랑은 배점상 무의미(10점 항목).

## 4. 배점 → 행동 매핑 (대상 시나리오)

| 배점 | 우리 대응 | 발표에서 보여줄 것 |
|---|---|---|
| 행정 적용 15 | B2G 우선순위+근거카드+인쇄물 | 실제 A4 출력물을 심사위원에게 배부 |
| 시민 체감 15 | 고령자 페르소나 + 편안 경로 | "1분 더 걷고, 앉아서 기다립니다" 한 문장 |

| 시연 가능성 20 | 라이브 데모 + 오프라인 폴백 | **와이파이 끄고 데모 이어가기** (준비된 퍼포먼스) |

| 결합 타당성 5 | ID/공간/이름 3중 조인 구조 | 파이프라인 다이어그램 1장 |

## 5. 무박2일 작업 우선순위

- **P0 (필수, 1일차 저녁까지):** 쉼표지수 계산 + B2G 우선순위 테이블/지도, B2C 편안 경로, 시간대 슬라이더
- **P1 (2일차 새벽):** 예산 시뮬레이터, 근거 카드 확장(POI), 도착 알림
- **P2 (버퍼):** 발표자료·A4 출력물·와이파이 오프 데모 리허설 — **P2는 절대 줄이지 말 것.** 시연 40점 중 20점이 여기서 갈림.

## 6. 발표 14분 구성(안)

1. (2분) 문제 — 폭염 속 그늘 4곳/1,890 정류장, 김OO 님 이야기
2. (4분) B2C 라이브 데모 — 편안 경로, 중간에 와이파이 오프 → 오프라인 계속 동작
3. (4분) B2G 라이브 데모 — 우선순위 → 근거 카드 → 예산 슬라이더 → A4 인쇄물 배부
4. (2분) 데이터 — 6종 결합 구조 1장 + 정직성 규칙("미확인은 미확인으로")
5. (2분) 확장 — 조사 목록이 데이터를 계속 채우는 선순환, 타 지자체 복제 가능성
