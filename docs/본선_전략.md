# 쉼표 정류장 — 본선 대상 전략

> 제품 정의는 `docs/제품_기획.md`, 화면 순서는 `docs/본선_스토리보드.md`를 따른다.

## 1. 기억시킬 문장

**버스가 언제 오는지는 기존 앱도 알려줍니다. 쉼표 정류장은 고령 이용자가 그 시간을 어떤 환경에서 기다려야 하는지 알려주고, 그 정보의 빈칸을 춘천시의 조사·개선 업무로 연결합니다.**

## 2. 경쟁 우위

| 심사축 | 우리의 증거 |
|---|---|
| 정책 실효성 | 조사→시설별 개선 검토→현장 공사 판단의 행정 절차 |
| 시민 체감 | 방향별 QR, 큰 글자, 음성 목적지, 대기시간과 시설을 한 화면에 표시 |
| 기술 구현 | 서로 다른 ID·좌표·시설 관측·시민 집계를 정류장 단위로 결합 |
| 데이터 타당성 | 양방향 합산 금지, 3상태, 출처·기준일, 결측·충돌 공개 |
| 확장성 | 여름 그늘에서 겨울 온열·야간 조명·BIT로 동일 구조 확장 |

## 3. 대상급 데모 한 사이클

1. 실제 방향별 정류장 QR을 촬영한다.
2. 현재 정류장명·번호·방면을 확인한다.
3. 목적지를 말한다.
4. 탈 버스·예상 대기시간과 현재 정류장 시설상태를 본다.
5. 시설정보가 다르면 구조화 제보를 남긴다.
6. 행정 화면에서 해당 정류장이 `시설 재확인 필요`로 나타난다.
7. 사람이 검수한 조사 결과를 반영한다.
8. 시민 화면의 시설상태와 행정 개선 검토 목록이 함께 바뀐다.

사전에는 스토리보드·스키마·데이터 목록만 준비하고, 핵심 연결과 최종 시연 기능은 본선 기간 커밋으로 명확히 남긴다.

## 4. 발표에서 분리할 값

| 값 | 정확한 이름 |
|---|---|
| 공공 승하차 | 표본기간 시간대별 공식 승차수요 |
| QR 스캔 | 방향별 QR 서비스 이용 |
| 목적지 선택 | 목적지 조회 의도 |
| 연령 선택 | 선택 응답자의 연령대 |
| 시설 제보 | 검수 전 시민 관측 |
| 조사 결과 | 사람이 확정한 시설 관측 |

서로 더해서 `고령자 승차량`이나 `수혜 인원`을 만들지 않는다.

## 5. 본선 전 허용 준비

- 1차 기획서와 데이터 원본 정리
- 정류장 ID 대응·품질 문제 목록
- 화면 와이어프레임과 스토리보드
- 데이터베이스·이벤트 스키마
- API·오픈소스·아이콘 조사
- 발표자료 목차와 빈 차트 틀
- 실제 사용자 테스트 계획

## 6. 본선 현장에 남길 핵심 개발

- 시민 핵심 화면의 최종 상태 연결
- 동의 기반 시민 이벤트 집계
- 행정 조사대상과 시설별 개선 검토 화면 연결
- 조사 데이터 반영 전후 시연
- 최종 데이터 분석 결과와 그래프 생성
- 접근성·오프라인·실기기 검증

## 7. 공격 질문

| 질문 | 답변 |
|---|---|
| 시민 신호가 실제 승하차입니까? | 아닙니다. 방향별 목적지 조회 신호이며 공식 승하차와 분리합니다. |
| 고령자 수는 어떻게 압니까? | 선택 연령대 응답만 집계하고 전체 고령 승객으로 일반화하지 않습니다. |
| 시설 미확인이 왜 설치 후보입니까? | 설치 후보가 아니라 조사 대상입니다. 없음이 확인된 뒤에만 개선 검토로 이동합니다. |
| 바로 공사할 수 있습니까? | 보도폭·도로점용·전력·예산은 별도 현장검토가 필요합니다. |
| 맞은편 의자를 이용하면 되지 않습니까? | 방향별 수요가 다르고 횡단 위험이 있어 같은 승차 정류장의 시설만 안내·평가합니다. |
