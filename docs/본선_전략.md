**배점이 말해주는 것:**

## 1. 방향성: B2C + B2G 투트랙, 두 가지 수정

**결론: 투트랙이 맞다. 단, 아래 두 가지를 수정해야 대상권이다.**

### 1-1. 투트랙이 맞는 이유 (배점 매핑)

| 트랙 | 직접 공략 배점 |
|---|---|
| B2C 시민앱 | 시민 체감 효과 15 + 서비스성 20 + 시연 가능성 일부 = **~40점대** |
| B2G 대시보드 | 행정 적용 가능성 15 + 분석 타당성 20 + 데이터 결합 구조 5 = **~40점대** |

한쪽만 하면 구조적으로 30점 안팎을 버리게 됨. 투트랙은 선택이 아니라 이 배점표의 요구사항. 두 트랙은 §4의 데이터 아키텍처("개인 데이터는 기기에, 통계 신호만 서버에")로 연결된다.

### 1-2. 수정 ① — B2G의 "실시간"은 버려라 (정직성 + 감점 리스크)

우리 승하차 데이터는 **2025.6.25~28 나흘치 정적 데이터**(엑셀 65535행 절단, 농촌 223곳 수요 미확인)다. 이걸로 "실시간 수요 분석"을 주장하면:
- 심사위원(데이터 전문가 포함 가능)이 "데이터 출처가 뭐냐"고 묻는 순간 무너짐 — **분석 타당성 20점 직격**.
- 우리 팀 정체성인 정직성 규칙(근거 없는 주장 금지)과도 충돌.

**재프레임:** "실시간 예산 분석" → **"시간대별 수요 패턴 기반 투자 우선순위"**
- 시간대별(byHour 24칸) 수요 × 시설 결핍(3상태) × 노선 수를 결합한 **정적이지만 방어 가능한 우선순위 모델**.
- "실시간"이 필요하면 유일하게 진짜 실시간인 **TAGO 도착정보**만 그 단어를 쓰고, 나머지는 "최신 집계 기준 YYYY.MM" 배지로 표기.
- 오히려 이 정직함 자체를 발표 무기로: *"저희는 없는 데이터를 실시간이라고 부르지 않습니다. 미확인은 미확인으로 표시합니다"* — 행정 적용 가능성 심사에서 신뢰로 환산됨.

### 1-3. 수정 ② — 무박2일 범위 통제

본선은 사실상 **개발 가용 시간 ~15시간**(1일차 14:30~24:00 + 2일차 00:00~07:30). 신기능은 §6의 우선순위 P0~P1까지만. 새 데이터 파이프라인 재설계 금지 — stops.json 계약은 그대로 두고 위에 얹는다.

## 2. B2C 앱 — "집에서 목적지까지, 앉아서 기다리는 길"

### 2-1. 타깃 세대: 고령자(65+) 단일 타깃 유지

- 근거 제시용 스토리: 춘천은 고령화 진행 지역, 폭염 속 그늘·의자 없는 정류장 대기가 실제 건강 리스크. "시민 체감 효과 15점"은 **누가, 언제, 얼마나 체감하는지**를 구체적 인물(페르소나)로 답해야 함.
- 타깃을 넓히면(전 세대) 화면이 복잡해지고 서비스성 20점("특정 사용자 관점의 편의성")이 오히려 깎임. **한 세대에 깊게**가 정답.
- 페르소나 예: *"후평동 78세 김OO 님, 병원 가는 길. 스마트폰은 카카오톡만 쓰심."*

### 2-2. 핵심 가치 제안

> "주변 쉼터·정류장 → 도착지까지의 연결성 → **가장 편하고 안락하게** 가는 방법"

한 문장 기능으로: **"편안 경로(쉼표 경로)" — 가장 빠른 길이 아니라, 그늘·의자가 있는 정류장을 경유해 서서 기다리는 시간을 최소화하는 길.**

### 2-3. 기능 구성 (기존 자산 → 본선 추가)

**이미 있음 (재사용, 발표 때 "구현 완료"로 시연):**
- 지도 + 시설 필터(그늘/의자/조명/도착안내기 3상태), 시설 근거 배지("벤치대장 기준", "로드뷰 확인 YYYY.MM")
- 정류장 카드(경유 노선, 시간대 수요), 즐겨찾기, 대체 정류장 힌트(AltStopHint)
- 도보 경로(OSRM WalkLayer) + 여정 계획(planTrip)
- TAGO 실시간 도착 + 배차간격 폴백, PWA 오프라인
- URL 공유 대리등록(`?fav=` 화이트리스트 검증 + XSS 방어), 로컬 QR 생성(네트워크 불필요), A4 인쇄 포스터

**본선에서 추가 (P0 = 필수):**
1. **편안 경로 추천** — 출발지→목적지 입력 시 후보 정류장들을 "편안 점수"로 정렬해 제시. 점수 = 의자(확인) + 그늘(확인) + 야간이면 조명 + 도보거리 패널티. 기존 planTrip/altStop 로직 확장이라 15시간 내 가능. **온디바이스 규칙 기반으로 충분** — 개인 데이터에 LLM을 붙이지 않는다(오프라인·프라이버시·속도 모두 유리).
2. **정류장 QR 즐겨찾기 등록** — 정류장에 붙인 QR을 찍으면 **설치도 가입도 없이** 그 정류장이 내 폰에 등록. 부품 전부 존재(`?fav=` 딥링크 → `ImportOnLoad` 자동 병합 → `qr.ts` → `PrintPoster`), 남은 작업 반나절 미만: ① 정류장 카드에 "이 정류장 QR" 버튼(`buildShareUrl([stop.id])` 재사용) ② 인쇄 포스터에 QR 삽입 ③ 등록 직후 확인 문구("○○정류장을 즐겨찾기에 넣었어요") ④ **배포 URL 선고정**(인쇄물 QR은 origin이 박제됨). **로그인은 만들지 않는다** — §4-1 참조. 인쇄 포스터+QR은 행정이 실제 배포할 수 있는 산출물이라 B2G(행정 적용 15점)와도 연결.
3. **"지금 기다리기 좋은 곳"** — 현재 시각 + 폭염 시간대(예: 13~16시)면 그늘 있는 정류장을 우선 표시. 시간대 수요(byHour)와 결합해 "지금 붐비는 정류장 → 옆 정류장 제안".
4. **초대형 글씨 모드 마감** — 기존 고령자 UI 기반 위에서 글자 크기 토글·아이콘+한글 병기 일관성 점검(디자인-a11y 에이전트 활용). ※ 음성(TTS)은 신규 구현이므로 P-거절.

**P1 (시간 남으면):**
- 도착 알림(진동/소리) — "버스 3분 전, 이제 일어나세요"
- 보호자 공유 강화: 여정을 QR로 자녀에게 전송 (`?trip=출발id,도착id` 확장, §4-2 Tier 2)

**만들지 말 것 (P-거절):** 회원가입/로그인, 챗봇, 전 세대용 모드 분기, 실내지도, 원격 즐겨찾기 동기화(§4-2 Tier 4). 전부 15시간을 잡아먹고 배점에 안 붙음.

### 2-4. 화면 구성 (고령자 UX 원칙 포함)

```
[홈] ──── 큰 버튼 2개가 전부
 ├─ "🚌 버스 타러 가기"  → 가까운 정류장 1곳을 바로 크게 (선택지 최소화)
 └─ "⭐ 자주 가는 곳"     → 즐겨찾기 목록 (병원/시장/집)

[정류장 카드] ─ 한 화면 = 한 정보 원칙
 ├─ 도착정보: "16번 버스, 약 5분" (숫자 최대 크기, 실시간/배차폴백 구분 배지)
 ├─ 시설: 아이콘+한글 병기 "🪑 의자 있음 (벤치대장 기준)" / "❓ 그늘 미확인"
 └─ 버튼: "여기로 가는 길 안내" / "이 정류장 QR" (보호자·포스터용)

[편안 경로 (신규)] ─ 지도 최소화, 문장 중심
 ├─ "① 집에서 4분 걸어서 → ② ㅇㅇ정류장(의자·그늘 있음)에서 앉아서 기다리기
 │    → ③ 16번 버스 → ④ 병원 앞 하차"
 ├─ 비교 한 줄: "가장 가까운 정류장보다 1분 더 걷지만, 앉아서 기다립니다"
 └─ [크게 보기] [자녀에게 보내기(QR)]

[즐겨찾기] ─ 이름 크게, 사진/이모지, 탭 1번이면 경로 시작
 └─ QR 스캔 유입 시: "○○정류장을 즐겨찾기에 넣었어요" 확인 문구
```

**고령자 UX 체크리스트 (서비스성 20점 방어):**
- 글자 최소 20px/버튼 56px 이상, WCAG AA 대비, 터치 타깃 간격
- 화면당 의사결정 1개, 뒤로가기는 항상 같은 위치
- 전문용어 금지: "정류소 ID" ❌ → "정류장 번호판의 숫자" ⭕
- 모든 상태에 문장 폴백: 인터넷 끊김 → "지금은 저장된 정보로 보여드려요 (어제 기준)"

## 3. B2G 웹 — "예산을 어디에 먼저 쓸지, 근거와 함께"

### 3-1. 공무원(수요자)이 실제로 원하는 것

담당 부서를 상상하라: 교통행정과·도시재생과 주무관. 이들이 의회·상급자에게 보고할 때 필요한 것은 화려한 지도보다 **"방어 가능한 근거"**다.

| 공무원의 질문 | 우리가 주는 답 |
|---|---|
| "내년 벤치/그늘막 예산 N개, 어디부터?" | **투자 우선순위 랭킹** (수요×결핍 점수, 정렬된 목록) |
| "왜 거기가 1순위인가?" (의회 답변용) | **근거 카드** — 시간대 수요 그래프 + 시설 미비 현황 + 데이터 출처 명기 |
| "노인이 많이 쓰는 곳인가?" | 병원·복지관·시장 인접 정류장 가중치 (POI 결합) |
| "실제 시민 반응은?" | **시민 신호 히트맵** — QR 스캔·즐겨찾기 익명 카운트 (§4-2 Tier 3) |
| "이거 보고서에 붙일 수 있나?" | **CSV 내보내기(구현 완료) + A4 인쇄(구현 완료)** — 한글(hwp) 문화권 행정엔 인쇄물이 곧 채택 |
| "데이터가 오래된 거 아닌가?" | 모든 지표에 기준일 배지 + "미확인 1,886곳 → 조사 필요 목록 자체가 산출물" |

### 3-2. 핵심 지표: 쉼표지수 (투자 우선순위 점수)

```
쉼표지수 = 수요 가중치 (시간대별 승차, 특히 한낮 11~16시 폭염 시간대)
         × 시설 결핍도 (의자·그늘 '있음 확인' 없을수록 ↑)
         × 이용 취약 가중치 (병원/복지시설/시장 반경 300m 인접)
         × 노선 수 (환승·대기 발생량 프록시)
   보조: + 시민 신호 (QR 스캔·즐겨찾기 익명 카운트 — 수요 데이터에 없는
           농촌 223곳의 공백을 시민 신호로 보완)
```

- 수요·결핍·노선수는 **이미 보유한 데이터**(stops.json)로 계산 가능. **취약 가중치(POI)는 아직 미확보 데이터**(§3-5)에 의존 → **옵션 항으로 설계**(POI 확보 시 가중, 미확보 시 1.0 고정)해서 P0가 데이터 확보에 막히지 않게 한다.
- "미확인" 시설은 결핍으로 단정하지 않고 **"조사 우선순위" 별도 트랙**으로 분리 — 정직성 유지가 곧 행정 신뢰성.
- 동선 이야기는 즐겨찾기가 아니라 **기존 시간대별 승하차 데이터로만** 한다(즐겨찾기는 "정류장 목록"이지 "이동"이 아님 — 근거 없는 주장 금지).

### 3-3. 대시보드 산출물 5종

1. **우선순위 지도** — 쉼표지수 히트맵/등급 마커, 시간대 슬라이더(아침/한낮/저녁 수요 전환)
2. **Top-N 리스트 + 근거 카드**(EvidenceCard 확장) — 카드 문구 형식(가상 예시): "인성병원 앞: 한낮 승차 4위(양방향 합산), 의자 미확인, 노선 N개, 병원 도보 N분". ※ "한낮 승차 4위"만 실측(데이터_현황 §3), 노선 수·도보 시간은 구현 시 실계산으로 채울 것. 현행 대시보드 실측 TOP은 중앙로입구·명동입구(QA_체크리스트) — 쉼표지수 도입 후 순위는 달라질 수 있음.
3. **예산 시뮬레이터(간단 버전)** — "벤치 1개 80만 원 × 예산 3,000만 원 → 37곳 커버, 추정 수혜 승차 N건/일" 슬라이더 1개면 충분. 심사위원 앞에서 움직이는 숫자는 강력함.
4. **시민 신호 히트맵 + AI 브리핑** — 익명 카운트(§4-2 Tier 3)를 순위·히트맵으로 표시하고, AI가 집계 통계를 자연어 보고서 초안으로 번역("이번 주 즐겨찾기 급증: ○○정류장, 의자 미확인 — 신규 수요 후보"). 이상 신호 탐지는 임계값 규칙으로 시작, AI는 해석문 담당. **행사장 네트워크 대비 사전 생성본 폴백 필수.** 데모 기간 수집량이 적으므로 "모의 누적 데이터" 표기 또는 팀원 기기 실데이터만 사용(정직성 규칙).
5. **조사 필요 목록** — 미확인 1,886곳 중 수요 상위 정류장부터 로드뷰/현장조사 우선순위 CSV. *"이 앱은 완성된 답이 아니라 행정이 계속 쓰는 도구"* 라는 확장성(15점) 서사.

## 4. 데이터 아키텍처 — "개인 데이터는 기기에, 통계 신호만 서버에"

B2C에서 데이터를 모아 B2G에서 쓰되 정체성을 지키는 방법. 핵심 분리: **"개인의 데이터"(누가 무엇을 즐겨찾았나)와 "통계 신호"(어느 정류장이 몇 번 즐겨찾아졌나)는 다르다.** B2G에 필요한 건 후자뿐이며, 후자는 개인정보가 아니다.

### 4-1. "개인정보 미수집"이 정체성인 이유

| 이유 | 내용 |
|---|---|
| ① 타깃 특성 | 고령자에게 회원가입·로그인은 최대 이탈 장벽. "설치도 가입도 없이 QR 찍으면 끝"이 서비스성(20점)의 핵심 |
| ② 아키텍처 | 서버 없는 정적 PWA(오프라인 우선)라 수집할 서버가 애초에 없음 — 미수집과 오프라인 강점은 한 몸 |
| ③ 법적 부담 | 개인정보보호법상 수집 시 동의·처리방침·안전조치·파기 의무 발생. 특히 고령자의 집·병원 이동패턴은 민감 정보 — 해커톤 팀이 질 수 없는 책임 |
| ④ 행정 신뢰 | 시가 채택할 도구에 개인정보 리스크가 0이라는 것 자체가 채택 장벽 제거(행정 적용 15점). 보안점검 문서에 "개인정보 미수집 — 통과"로 검증·문서화됨 |
| ⑤ 심사 방어 | "개인정보는요?"라는 단골 질문을 원천 차단 |

**미수집 ≠ 데이터 포기.** 아래 계층 설계로 B2C→B2G 데이터 흐름을 만든다.

### 4-2. 계층별 저장소 설계 (localStorage vs 클라우드 결론)

```
Tier 1  개인 데이터        → localStorage (현행 유지)
        즐겨찾기 id, 글자크기 설정. 기기 소유, 오프라인 완전 동작, 동의 불필요.

Tier 2  사람→사람 공유     → URL/QR 페이로드 (현행 + 확장)
        데이터가 서버가 아니라 "링크 안에" 실려 이동. ?fav=id1,id2 (구현 완료)
        확장: ?trip=출발id,도착id — 자주 가는 경로도 같은 방식으로 전달 가능.
        자녀/요양보호사가 만들어 QR로 주면 부모 폰 localStorage에 병합(대리등록).

Tier 3  익명 집계 신호     → 클라우드 (신설, 옵트인, B2G 원료)
        스키마: { stop_id, event: fav_add|qr_scan|trip_plan, hour: 0-23, date }
        금지: 사용자 식별자·기기ID·IP·이름·좌표이력. 카운트 증가만.
        구현: Supabase 테이블 1개 or Cloudflare Worker+KV. 전송 실패는 조용히 무시(오프라인 원칙).
        소비: B2G 대시보드 히트맵·순위 + AI 브리핑 (§3-3 ④).

Tier 4  원격 동기화        → 하지 않음 (본선 범위 밖, 필요시 로드맵으로만 발표)
        "자녀가 원격으로 부모 즐겨찾기를 갱신"은 페어링 코드+클라우드 KV로 가능하지만
        (랜덤 토큰 키, 이름·전화 없음, TTL), 계정 없는 유사-계정이라 프라이버시 설명 부담↑.
        QR 대리등록으로 데모 시나리오가 이미 성립하므로 무박2일에 넣지 않는다.
```

**자주 나오는 두 가지 오해:**
- *"다른 사람(자녀·요양보호사)이 저장해둔 즐겨찾기를 가져오려면 로컬로는 안 되지 않나?"* → **이미 된다.** 공유 URL/QR에 정류장 id 목록이 실려 서버 없이 기기→기기로 이동한다(스냅샷 방식). 안 되는 것은 "원격 실시간 동기화"뿐이며, 그건 Tier 4로 의도적으로 제외한다.
- *"수집엔 클라우드 DB가 효율적?"* → **집계엔 맞고, 개인 데이터엔 틀리다.** 개인 즐겨찾기까지 클라우드로 옮기는 순간: 로그인 필요(고령자 장벽) + 개인정보 처리자 의무 발생 + 오프라인 PWA 강점 상실 + 무박2일에 auth/보안 구현 부담. 통계 신호만 클라우드로 보내면 클라우드의 수집 효율과 미수집 정체성을 동시에 가진다.

### 4-3. 제안 기능 타당성 검토 결론 (2026-07-16)

| 제안 | 판정 | 요지 |
|---|---|---|
| 정류장 QR → 스캔 즉시 즐겨찾기 | ✅ 가능, 거의 완성 | 부품 전부 존재, 반나절 미만 (§2-3 P0-2). 단 "로그인"은 만들지 않음 |
| 즐겨찾기 취합 → AI 동선 분석 | ⚠️ 원안 불가, 재설계로 가능 | ① 서버가 없어 취합 불가 ② 개인 이동패턴 수집 = 정체성 붕괴 + 심사 공격 포인트 ③ 즐겨찾기로는 동선 도출 불가(근거 없는 주장). → **익명 카운트(Tier 3) + 시민 신호 히트맵 + AI 브리핑**(§3-3 ④)으로 재설계 |

### 4-4. AI의 자리 (집계 위에서만)

| 위치 | 용도 | 비고 |
|---|---|---|
| B2G 어드민 | 집계 통계 → 자연어 브리핑 생성 | 사전 생성본 폴백 필수 |
| B2G 어드민 | 이상 신호 탐지(fav 급증 = 신규 수요 후보) | 임계값 규칙으로 시작, AI는 해석문 담당 |
| B2C 앱 | 편안 경로 점수·추천은 온디바이스 규칙 기반으로 충분 | 개인 데이터에 LLM 붙이지 않음 |

발표 문장: *"개인 데이터는 어르신 폰 밖으로 나가지 않습니다. 시청 대시보드가 받는 것은 '어느 정류장이 몇 번'이라는 숫자뿐이고, AI는 그 숫자를 공무원의 언어로 번역합니다."*

## 4-5. ⚠️ 개발 착수 전 필수 선행 결정 3가지 (Opus 핸드오프 게이트)

기존 문서(specs/plans/QA)와 이 전략 사이에 **충돌 3건**이 있다. 해소 없이 개발을 맡기면 기존 Global Constraints와 모순에 빠진다.

1. **합성 점수 금지 규칙 개정 (팀 합의 필요 — 최우선).**
   v1 설계·계획·QA 전부에 "대시보드 합성 점수(가중치 점수) 표시 금지, 조건 필터만"이 하드 룰로 박혀 있다(spec §5 B-1, plan Global Constraints, QA 표현규칙 검사). 그런데 **쉼표지수와 편안 점수는 정확히 합성 점수다.**
   → 제안 개정안: *"합성 점수는 ①산식 전체 공개 ②구성 요소별 실측값 병기 ③출처·기준일 배지 동반 시에만 화면 표시 허용"*. 이 개정은 stops.json 계약과 같은 급의 팀 합의 사항 — 개발 시작 전에 합의하고 spec/plan/QA 문서의 해당 문구를 함께 갱신할 것.

2. **AI 브리핑의 API 키 위치 확정.**
   보안점검 §1 잔여 리스크: `VITE_` 환경변수는 **빌드 시 클라이언트 번들에 평문 인라인**된다(TAGO에서 실측 확인). LLM API 키를 같은 방식으로 넣으면 과금 키가 공개 배포됨 — TAGO보다 피해가 크다.
   → 규칙: **클라이언트 번들에 LLM 키 절대 금지.** (a) 데모는 사전 생성 정적 JSON(브리핑 결과물)만 배포, (b) 라이브 생성이 필요하면 Tier 3 서버리스(익명 카운터와 같은 곳)에서 프록시 호출.

3. **"서버 0개" 문구·CSP·보안점검 갱신 태스크 포함.**
   Tier 3 카운터/AI 프록시를 도입하면: ① spec의 "서버 0개" 정체성 문구를 "개인정보 서버 0"으로 조정 ② `app/index.html` CSP `connect-src`에 신규 엔드포인트 추가(누락 시 브라우저가 조용히 차단 — 기능이 이유 없이 실패하는 함정) ③ 보안점검.md "외부 전송 2곳뿐" 항목 재검증. 세 가지를 구현 태스크에 명시적으로 포함할 것. 카운터는 옵트인·실패 무시라 오프라인 데모 하드 제약은 유지됨.
