## 1. 서비스 한 줄 정의

**"버스를 기다리는 자리가 살 만한지"를 알려주는 서비스.**

- **시민용(B2C)**: 이 정류장에 그늘·의자·도착안내기·야간조명이 있는지 지도로 보여주는 모바일 웹앱. 페르소나는 폭염·한파에 취약한 **고령자**.
- **행정용(B2G)**: 춘천시가 다음 유개식 승강장(쉘터)을 어느 정류장에 세울지 후보 목록을 뽑아주는 대시보드. (춘천시는 유개식 승강장 확대 사업 진행 중 — 2024년 발표 기준 264곳 추가 계획, 출처: 전자신문 2024.7.3)

본선 평가기준: **기술구현 40 / 정책실효 30 / 서비스성 20 / 데이터활용 10** → 실제로 돌아가는 데모의 완성도가 가장 큰 배점. 화면 완성도에 리소스를 집중할 것.

## 3. 산출물 구성 (3개)

```
[C] 데이터 파이프라인 (Python)  →  통합 정류장 JSON/CSV
        ├→ [A] 시민용 모바일 웹앱 "쉼표 정류장"   ← 데모 주인공
        └→ [B] 행정용 대시보드 "설치 우선순위"    ← 정책실효 담당
```

### [A] 시민용 모바일 웹앱 — 데모 주인공
설치 없는 모바일 웹(반응형, 가능하면 PWA). 고령자 기준 설계가 원칙: 기본 글자 크기부터 크게, 정보는 아이콘+한글 병기, 화면당 행동 1개.

### [B] 행정용 대시보드
쉘터 설치 후보를 조건 필터로 추리는 관리자 화면. 데모 클라이맥스: 필터 적용 → "설치 후보 TOP 10" 표 → CSV 내보내기 버튼 → "이대로 담당 부서에 드리면 됩니다."

### [C] 데이터 파이프라인
원본 CSV들을 정제·결합해 A와 B가 공통으로 읽는 정적 데이터 파일을 생성. 라이브 DB 불필요 — 해커톤장 네트워크를 신뢰하지 말 것 (TAGO 도착정보만 예외적으로 실시간, 실패 시 배차간격 캐시로 폴백).

## 4. 기능 명세 (우선순위별)

### P0 — 이것 없으면 데모 실패

**A-1. 첫 화면: 지도 + 내 주변** (민수쌤 피드백의 핵심)
- 접속 즉시: 현재 위치(권한 거부 시 춘천 중심가 기본값) 주변 지도 + 정류장 마커.
- 마커 자체가 시설 요약을 표현 (예: 그늘 있음 = 초록, 미확인 = 회색).
- 가장 가까운 정류장이 자동 선택되어 하단 카드로 뜸: 정류장명 + 그늘/의자/조명/도착안내기 4개 아이콘 + 각각 "있음(확인)/없음(확인)/미확인".

**A-2. 3상태 표기 (전 화면 공통 규칙)**
- 모든 시설 정보는 반드시 `있음(확인) / 없음(확인) / 미확인` 3가지로만 표시. 조사 안 한 곳을 "없음"으로 표시하는 코드 경로가 존재하면 안 됨 (타입 수준에서 막을 것).
- 확인된 정보에는 "로드뷰 확인 (촬영 시점 YYYY.MM)" 배지. "현장 확인"이라는 문구는 사용 금지.

**A-3. 즐겨찾기 + 보호자 대리 등록**
- 자주 타는 정류장을 별표로 저장 (localStorage, 로그인 없음).
- 대리 등록: 자녀가 자기 폰에서 정류장을 고르고 "공유하기" → URL(쿼리스트링에 정류장 ID 목록) 생성 → 부모 폰에서 그 링크를 열면 즐겨찾기가 자동 등록되고 큰 글씨 홈 화면으로 시작. 서버·계정 불필요.
- 즐겨찾기 화면은 초대형 카드 목록: "○○ 정류장 — 그늘 있음, 의자 있음, 버스 7분 후".

**B-1. 조건 필터 + 후보 TOP N**
- 필터 예: `한낮(11~16시) 승차 상위 25% AND 그늘막 확인 안 됨 AND 유개식 승강장 아님`.
- 결과 표: 정류장명, ID, 승차량, 시설 현황, 근거 요약. CSV 다운로드 버튼.
- **점수 산식 금지.** "위험점수 78점" 같은 합성 점수를 만들지 말 것 — 가중치를 방어할 수 없다. 조건 자체가 설명이 되는 필터만 사용.
- 계절 프리셋 4종 (여름: 그늘 기준 / 겨울: 방풍·온열의자 기준 / 봄가을: 기본). 프리셋 = 저장된 필터 조합일 뿐, 별도 로직 아님.

**C-1. 통합 정류장 마스터 생성**
- 승하차 CSV의 7자리 정류장아이디(424xxxx)를 기준 키로 모든 데이터를 ID 조인. 이름 조인 금지 (동일 명칭 양방향 정류장 존재 — 예: 명동입구 2개).
- 단계적 결합 (하나 실패해도 데모 가능해야 함):
  1. 좌표 기반 기본 지도 (실패 불가 수준)
  2. 승하차 ID 조인 (시간대별 승차량)
  3. 그늘막·벤치·가로등 결합
  4. 로드뷰 조사 결과 (팀이 150곳 수동 조사 예정 — 입력용 스프레드시트 양식을 파이프라인이 읽을 수 있게 먼저 정의해서 팀에 줄 것)

### P1 — 데모를 강하게 만드는 것

**A-4. 버스 도착 정보**: TAGO API(도착정보 15098530)로 선택 정류장의 도착 예정 표시. API 실패 시 "배차간격 약 N분" 캐시 폴백. 데모 중 스피너가 도는 상황이 없어야 함.

- 예상 질문 "더운데 누가 옆 정류장까지 가나요?"에 대한 답이 코드에 구현되어 있는 형태여야 함: 곧 버스가 오면 안 뜬다.

**A-6. A4 안내문 인쇄 (앱 밖 경로)**: 정류장 선택 → "안내문 인쇄" → 큰 글씨 A4 인쇄용 화면(브라우저 인쇄로 충분). 경로당·행정복지센터·정류장 게시판 부착용. 고령자가 앱을 안 써도 혜택을 받는 경로라서 발표에서 중요.

## 6. 기술 스택 제안

3인 팀, 데이터 분석 교육 수료생(파이썬 중심), 해커톤 데모가 목적이라는 전제.

- **[C] 파이프라인**: Python + pandas. 산출물은 `stops.json` 하나(정류장별 좌표·시설 3상태·시간대 승차량·경유 노선). 재실행 가능한 단일 스크립트로.
- **[A] 시민용**: 정적 웹앱 (프레임워크는 Vite + 바닐라 or React 중 구현자가 편한 쪽). 지도는 **Kakao Maps JS SDK**(국내 정류장 데모에 로드뷰·한국 지도 품질 이점, 무료 키) — 키 발급이 막히면 Leaflet + OSM 폴백. 서버 없이 정적 호스팅으로 시연.
- **[B] 행정용**: **Streamlit** (팀이 발표 중 직접 조작하기 쉽고 CSV 내보내기·표·필터가 공짜). 시민용과 굳이 같은 스택일 필요 없음.
- 배포: 로컬 실행이 기본, 시연 백업으로 정적 빌드 + 휴대폰 실기기 확인.

이 선택은 제안이며 구현자가 더 빠른 대안이 있으면 바꿔도 된다. 단, "해커톤장 네트워크 없이도 돌아간다"는 제약만은 유지할 것.

## 7. 표현·주장 규칙 (발표·화면 문구 공통, 위반 시 신뢰 붕괴)

2. 배지는 "로드뷰 확인 (촬영 시점 표기)". "현장 확인" 금지.
3. 쉘터 사업 수치는 "2024년 발표 기준 264곳 추가 계획"으로만. "지금 264곳 남았다" 단정 금지.
4. 인성병원 순위를 언급하면 반드시 "한낮 승차 4위 (같은 이름의 양방향 정류장 합산 기준)" 병기. ID 단위로는 6위.
5. "시간대별 승하차 데이터는 춘천만의 희귀 데이터" — **금지된 주장** (반증됨).
6. 4일 표본 방어: "날짜별 승차 순위 상관 0.91~0.93, 상위 20곳 중 15곳 이상 일치" 실측을 근거로 사용.
7. 승하차 데이터는 2025.11 노선 조정 이전 자료 → 정류장 단위 수요로만 쓰고 노선 단위 주장에 쓰지 않는다.
8. 합성 점수(가중치 점수) 화면 표시 금지 — 조건 필터로만.

## 8. 완료 기준 (데모 시나리오가 곧 정의)

발표 데모가 다음 시나리오를 네트워크 불안 상태에서도 통과하면 완료:

1. 휴대폰에서 접속 → 3초 내 지도 + 가장 가까운 정류장 카드 (조작 0회).
2. 정류장 카드에서 그늘·의자·조명·안내기 3상태 + 로드뷰 배지 확인.
3. "자녀 폰 → 공유 링크 → 부모 폰 즐겨찾기 등록" 시연 (폰 2대 or 폰+PC).
4. (P1) 배차 30분 노선 상황에서 대안 정류장 제안이 뜨고, 곧 도착 상황에선 안 뜨는 것 시연.
5. (P1) 정류장 안내문 A4 인쇄 미리보기.
6. 대시보드에서 여름 프리셋 → TOP 10 표 → CSV 다운로드 → "이대로 담당 부서에 드리면 됩니다" 멘트.

## 9. 구현자(Opus)에게

- 기능을 늘리자는 유혹이 오면 2절의 "많은 기능 필요 없음" 피드백이 항상 이긴다. P0의 화면 완성도 > P1 추가 > P2.
- 데이터 정제에서 막히는 부분(ID 체계 불일치 등)이 나오면 우회하지 말고 3상태의 "미확인"으로 정직하게 처리 — 그게 이 서비스의 정체성이다.
- 팀원 3인은 개발 숙련도가 높지 않다. 실행 방법은 README에 명령어 복붙 수준으로, 로드뷰 조사 입력은 스프레드시트 양식으로 만들어 줄 것.
- 배경 맥락이 더 필요하면 `fable\기획서_쉬운설명.md`(팀 내부 설명서)와 제출 기획서 docx를 참고.
